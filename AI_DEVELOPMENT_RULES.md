# Korea-SafeMate v6 AI Development Rules

이 프로젝트는 Korea-SafeMate v6이다.
v3를 기준으로 복제한 업그레이드 버전이며, 기존 기능 안정성을 유지하면서 점진적으로 개선한다.

**우선순위:** 제품·데이터·스택 원칙은 [FOUNDATION.md](./FOUNDATION.md)가 우선한다. 본 문서는 AI·인간 협업 **절차·UX·보고**를 보완한다. 충돌 시 Foundation을 따른다.

## 역할

AI는 단순 코드 생성자가 아니라, PM/UX/개발 기준을 따르는 구현 파트너로 작업한다.

## 기본 원칙

1. 기존 기능을 임의로 삭제하지 않는다.
2. 작업 전 Plan을 작성한다.
3. Plan 범위 안에서만 구현한다.
4. 작업 후 Check를 수행한다.
5. 변경 내용은 문서화한다. **`docs/dev-log.md`** 에 작업 로그를 남긴다(형식은 동 파일 및 아래「작업 로그」절).
6. 검증하지 않은 것은 검증했다고 말하지 않는다.
7. 대시보드 설정이 필요한 일은 코드로 해결한 것처럼 말하지 않는다.

## UX 원칙

1. 모바일 우선으로 설계한다.
2. Traveler / Guardian / Admin 역할이 명확해야 한다.
3. 사용자는 현재 화면에서 다음 행동을 이해할 수 있어야 한다.
4. 보이는 버튼은 반드시 동작해야 한다.
5. 동작하지 않는 UI는 남기지 않는다.
6. 로딩/빈 상태/에러 상태를 고려한다.
7. 지도, 스팟, 루트, 가디언 정보는 하나의 여정 맥락으로 연결한다.

## 개발 원칙

1. Supabase schema는 임의로 변경하지 않는다.
2. 인증/권한 로직은 최소 변경만 허용한다.
3. 환경변수는 하드코딩하지 않는다.
4. `.env.local`은 커밋하지 않는다.
5. 새 라이브러리는 사전 근거 없이 추가하지 않는다.
6. 타입 에러를 `any` 또는 `@ts-ignore`로 덮지 않는다.
7. mock은 mock임을 명확히 표시한다.
8. 실제 기능처럼 보이는 가짜 구현은 하지 않는다.

## 검증 원칙

`package.json`의 `scripts`를 확인하고 **실제 존재하는 명령어만** 실행한다. 이 저장소는 패키지 매니저로 **pnpm**을 사용한다([HARNESS.md](./HARNESS.md) §5).

가능하면 아래를 수행한다.

- `pnpm lint`
- `pnpm exec tsc --noEmit` *(루트에 `typecheck` 스크립트가 없을 때의 타입 검사)*
- `pnpm build`

브라우저 확인 항목:

- 콘솔 에러
- 주요 CTA 클릭
- 로그인/로그아웃
- 권한별 접근
- 모바일 레이아웃
- 빈 상태
- 에러 상태

## 보고 형식

작업 완료 후 아래 형식으로 보고한다.

1. 작업 목표
2. Plan 요약
3. 수정 파일
4. 파일별 변경 내용
5. 기능 연결 관계
6. 검증 결과
7. 미검증 항목
8. 남은 리스크
9. 다음 작업 제안
10. Git 상태

보고 형식의 요약·교차 참조는 [HARNESS.md](./HARNESS.md) §6과 맞춘다.

## 작업 로그(필수)

작업 완료 후 **반드시** [docs/dev-log.md](./docs/dev-log.md)에 기록한다. (과거 맥락은 루트 [DEV_LOG.md](./DEV_LOG.md)를 참고할 수 있으나, **신규 기록은 `docs/dev-log.md` 우선**.)

다음 형식을 따른다.

```markdown
## YYYY-MM-DD - [작업명]

### 목표
-

### 변경 파일
-

### 변경 내용
-

### 검증 결과
-

### 남은 이슈
-

### 다음 작업
-
```

**주의**

- 실제로 수정한 내용만 기록한다.
- 하지 않은 작업을 했다고 쓰지 않는다.
- 검증하지 않은 항목은 **미검증**으로 표시한다.
- 환경변수 값, API Key, 토큰은 기록하지 않는다.
